# Unicode·클립보드와 UTF-8

- **상태**: 검토 (정책 확정, 별도 UTF-8 복사 기능은 미구현)
- **관련 문서**: `doc/app-detailed-design.md` §2.3, `doc/upcoming-release-note.md`

## 고민한 점

- 입력창 텍스트를 복사하면 “UTF-8로 클립보드에 들어가나?”
- 웹브라우저·메모장이 UTF-8을 쓰는 것과 Windows 클립보드 표준이 같은지.

## 확인한 사실

| 계층 | 인코딩/포맷 |
|---|---|
| Rust `String` (앱 내부) | UTF-8 |
| Win32 `EDIT`, `W` API | UTF-16 |
| Windows 표준 텍스트 클립보드 | `CF_UNICODETEXT` (UTF-16LE) |
| 웹 페이지·메모장 **파일** 저장 | 종종 UTF-8 |
| 앱 간 복사/붙여넣기 | 보통 `CF_UNICODETEXT` 교환 |

웹이 UTF-8이어도 Ctrl+C 결과는 Windows에서 UTF-16 유니코드 텍스트인 경우가 일반적이다.

## 결정

- 기본 `Ctrl+C`는 **Windows 표준 유니코드 텍스트 복사**를 유지한다.
- 메모장, 브라우저, IDE, 메신저 등 일반 앱 붙여넣기 호환을 우선한다.
- UTF-8 **바이트**가 꼭 필요하면 기본 복사와 분리한 별도 기능으로 검토한다.
  - 예: UTF-8 파일로보내기
  - 예: 커스텀 클립보드 포맷 (호환 앱 제한적)

## 데이터 무결성

- 한글 문자열 자체는 Unicode이므로, 유니코드 지원 앱으로 복사·붙여넣기 시 깨지지 않아야 한다.
- ANSI 전용 프로그램이나 클립보드 처리 버그가 있는 앱은 예외.
